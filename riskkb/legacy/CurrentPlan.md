# Current Plan

## 当前判断

这个项目当前最关键、最难的部分，不是“生成最终答案”，也不是“把所有材料先塞进 RAG 再慢慢调 Prompt”。

真正的第一难点是：

**先把候选风险因子找对。**

也就是先把第一环节做成：

`症状 / 产品 / 人群 / 检验项 -> 候选风险因子 Top-K -> 证据打分 -> 允许不确定`

而不是做成：

`症状 -> 单一元凶`

因为一旦第一环节歪了，后面的：

- GB 依据
- 检测方法
- 生产环节
- 管控建议

都会一起歪掉。

## 当前已有的东西

### 1. GB 核心层

这是目前最硬的主依据层。

包括：

- `knowledge/standard_txt/`
- `knowledge/corpora/rag_corpus_standard_txt.jsonl`
- `knowledge/configs/gb2762_contaminant_limits.yaml`
- `knowledge/configs/gb_dairy_rules.yaml`

作用：

- 提供法规、限量、禁止添加、方法引用、标准互引
- 给最终答案提供可追溯的主依据

判断：

- 这层必须保留
- 但现在不应该再把主要精力继续放在“全面处理所有 GB 文本”上
- 继续大规模细抠 GB 全量正文，当前 ROI 不高

结论：

**GB 现在够做主依据层，不是当前最急的瓶颈。**

### 2. Config 规则层

这部分主要是我们自己抓取和整理的一些小规则，以及辅助推断规则。

包括：

- `risk_taxonomy.yaml`
- `stage_rules.yaml`
- 以及相关配置物化结果

作用：

- 症状到风险因子的映射
- 风险因子到生产环节的先验
- 一些显式规则和快速判断逻辑

判断：

- 这层很重要
- 但它不是法规事实层
- 只能作为辅助推断与排序，不能伪装成 GB 明文规定

### 3. PubMed / 文献扩充层

这部分是我们从 PubMed 和其他文献源抓到的内容。

作用：

- 提供症状、毒理、病原体、器官损伤、暴发特征等背景知识
- 用于扩充风险模式库
- 用于构建“症状 -> 风险因子”的已知模式识别层

判断：

- 这部分不是直接法规依据
- 但对第一环节非常关键
- 它决定系统能不能在症状出现时，把候选原因拉到合理的 Top-K

## 现在最困难的部分

当前最难做的，不是 GB，也不是多模态，而是：

**建立最小可用的“已知风险模式库”。**

也就是让系统在看到：

- 症状
- 人群
- 产品类型
- 检验项

之后，能先把候选风险因子排出来。

例如：

- 婴幼儿 + 奶粉 + 发热 / 脑膜炎倾向 -> `Cronobacter` 应该升高
- 冷藏乳制品 + 妊娠 / 免疫脆弱 + 发热 -> `Listeria` 应该排前
- 肾结石 / 泌尿系统损伤 + 奶粉 -> `melamine` 应该进入高优先候选

这一步如果没有做稳，后面的所有检索都会变成精致胡说。

## 当前核心策略

当前不追求“万能知识图谱”，也不追求一上来做成“大一统全能系统”。

当前最优先的策略是：

### 1. 先做小而硬的风险模式库

先建立一个高价值的已知风险因子库，而不是追求全覆盖。

建议起步规模：

- `50–150` 个高优先级风险因子

起步分类：

- 微生物类
- 化学污染物类
- 掺假 / 非法添加类

### 2. 第一环节做成候选排序，不做单点断言

输出应该是：

- 候选风险因子 Top-K
- 每个候选的触发依据
- 需要补查的证据
- 不确定说明

而不是让模型一口咬死“就是某个原因”。

### 3. 先做高 ROI 的数据治理，不做低 ROI 的全量文本精修

现在不应该继续把主要精力耗在：

- 全量 GB 逐条精修
- 全量大图谱抽取
- 为了做多模态而做多模态

现在应该优先做：

- 已知风险模式库
- 方法层 / 证据层的可调用治理
- 企业 / 产品 / 事件维度的数据收集

## 现在真的要继续处理 GB 语料吗？

结论：

**不是当前最高优先级。**

原因：

- GB 主依据层已经有可用基础
- 当前系统最缺的不是“再多一点 GB 正文”
- 而是“第一环节能不能先把候选原因找对”

所以当前不建议再把大量时间投入到：

- 继续扩充更多 GB 正文 chunk
- 继续为所有 GB 做更细粒度人工切分

除非后面遇到明确缺口，否则这不是最高 ROI。

一句话：

**GB 要保留，但当前主要精力不该继续砸在 GB 精加工上。**

## 接下来做什么

### 第一优先级：已知风险模式库

先建立一套最小可用的字段模板：

- `hazard_id`
- `hazard_name`
- `hazard_class`
- `product_domain`
- `symptoms`
- `vulnerable_group`
- `typical_products`
- `indicative_tests`
- `process_stage`
- `authority`
- `evidence_type`
- `source_url_or_id`
- `confidence`

目标：

- 让第一环节能先输出合理的 Top-K 候选风险因子

### 第二优先级：案例测试集

建立一个小规模但高质量的案例评测集。

每个案例至少标：

- 症状
- 人群
- 产品
- 检验项
- 正确风险因子
- GB 主依据
- 方法依据
- 可接受备选答案
- 不该召回的跨品类干扰项

不要先追求海量无标签语料，而是先做少量高质量评测集。

### 第三优先级：企业风险画像

企业风险画像不是“财报分析系统”。

更现实的是构建：

- 企业规模画像
- 供应链复杂度
- 产品风险结构
- 历史合规记录
- 标签宣传风险
- 渠道画像
- 信息透明度

对上市企业可以补财务信息，但财报不是主数据源。

### 第四优先级：多模态包装标签一致性审查

多模态不是第一优先级，但方向成立。

适合解决：

- 主宣称 vs 配料表
- 正面包装 vs 产品类别
- 宣称词 vs 标准属性
- 卖点图标 vs 法定信息

这部分可以后面作为新的并行方向推进。

## 数据策略

原则：

- 不求一开始全
- 但不能太少
- 要优先收集“高价值、可计算、可验证”的数据

### 优先收的数据

- 官方标准与方法
- 抽检通报
- 行政处罚
- 召回公告
- 官方风险评估
- 医学 / 毒理公开数据库
- 产品包装与标签
- 企业基础信息与许可信息

### 暂时不要过度投入的方向

- 全量 GB 精修
- 过早的大图谱工程
- 只靠 Prompt 的统一大 RAG
- 一开始就做复杂多模态大模型

## 关于 ClaudeCode Agent

接下来可以直接把 ClaudeCode 作为主 Agent 来配置。

定位：

- 高语义判断引擎
- 小步增量知识治理 Agent
- 案例抽取与模式整理 Agent

适合让 ClaudeCode 做：

- 逐批整理风险模式库
- 逐条补字段
- 清洗方法标准层
- 构建案例测试集
- 对新异常做候选归因

不适合让 ClaudeCode 独立做：

- 没有 schema 约束的大规模自动闭环
- 纯靠 Prompt 替代数据治理
- 一步到位做全量生产系统

## 当前最终结论

当前最有效、最高 ROI 的路线是：

1. 保留现有 GB 主依据层
2. 暂缓继续深加工全量 GB
3. 先做已知风险模式库
4. 建立小规模高质量案例测试集
5. 再做企业风险画像和多模态

一句话总结：

**现在不再继续围绕 GB 做低效精修，而是优先解决“第一环节候选原因定位”这个真正的命门。**
